# 国内酒店聚合搜索

用这个 skill 做中国国内酒店搜索时，不要只给单个平台结果。先判断位置，再多源搜索，再统一比价和评分。默认目标是找“性价比高且位置合理”的酒店，不是单纯找评分最高或价格最低。

## 输入要求

先从用户请求中提取：

- 目的地或 POI：城市、景区、商圈、游客中心、车站、机场等。
- 入住日期和离店日期：必须明确到具体日期。
- 入住人数：成人、儿童、房间数；缺失房间数时默认 1 间。
- 可选偏好：预算、星级/民宿/客栈、床型、早餐、停车、亲子、宠物、取消政策、排除区域。
- 出行方式：自驾/有车、公共交通、打车/包车、普通自由行未指定。

如果目的地、日期或人数缺失，先简短追问。不要猜日期，也不要用当前日期代替用户的入住日期。

## 作为旅行总控的住宿子流程

当 `travel-planning-cn` 或其他总控 skill 为多日行程调用本 skill 时，按“每天一个住宿任务”处理。总控应传入：

- 日期：当天入住、次日离店；多晚连住时明确连续日期。
- 人数和房间数：缺失房间数默认 1 间。
- 住宿锚点：当天终点城市/景区、次日出发方向、游客中心/停车场/车站/主路节点。
- 出行方式：自驾/公共交通/包车等，用于修正位置和设施权重。
- 行程偏好：预算、早餐、停车、亲子、海拔、避开县城/景区内不可进车区域等。

本 skill 只负责当天住宿区域和酒店推荐，不决定整条旅行路线。若发现总控给的住宿点明显不适合次日路线、无可订房源或价格异常，应在结论中标记“需总控调整路线/住宿点”，而不是擅自改全程行程。

返回给总控的每个推荐酒店都应尽量包含详情页或预订页链接。链接优先级：携程详情页、同程详情页、飞猪/FlyAI 详情页、来源搜索结果页。不能确认是同一家酒店时不要硬配链接；用 `需复核` 标注并保留可用的平台搜索页。

## Preflight

执行搜索前做来源健康检查，并先修复可修复的主来源问题。只输出“已配置/缺失/不可调用/需登录/已修复/空结果”和失败原因，不回显完整 key。

检查项：

- 高德地图 MCP 是否可用，优先使用 `maps_geo`、`maps_around_search`、`maps_distance`、`maps_direction_driving`、`maps_direction_walking`、`maps_direction_transit_integrated`。
- FlyAI 是否可用：优先检查 `flyai-cli` MCP 或 `flyai` CLI；检查 `FLYAI_API_KEY` 是否在环境变量、`~/.config/mcp/secrets.env` 或 `~/.flyai/config.json` 中。
  - 如果 CLI 缺失，先尝试 `npm i -g @fly-ai/flyai-cli` 并运行 `flyai --help`、一次轻量 `keyword-search` 或 `search-hotel` 验证。
  - 如果 `flyai-http` 静态 HTTP MCP 返回 401/403，但 `flyai-cli` MCP 或 CLI 可用，记录为“HTTP MCP auth failed, CLI path available”，并使用 CLI 路径。
- AIGoHotel MCP 是否配置且工具可调用。
  - 如果当前工具层未暴露但 Codex/Cursor config 有 `aigohotel`，先用 Streamable HTTP JSON-RPC 或 `scripts/aigohotel-mcp.mjs tools-list` 验证；能列出 `searchHotels`/`getHotelDetail` 时视为“可调用（HTTP fallback）”。
  - 搜索返回空酒店列表是来源“空结果”，不是环境不可用；需要换城市/区县/景点/详细地址等 placeType 重试后再记录。
- 同程程心 skill 是否存在：优先从当前会话技能列表或 `CHENGXIN_SKILL_HOME` 定位，再检查用户目录下的 `.agents/skills/chengxin`；确认 `scripts/hotel-query.js` 和 `CHENGXIN_API_KEY`。
- 携程问道 skill 是否存在：优先从当前会话技能列表或 `WENDAO_SKILL_HOME` 定位，再检查用户目录下的 `.agents/skills/wendao-skill`；检查 `WENDAO_API_KEY`。
- Browser 是否可用，用于同程和携程网页端真实搜索；同时检查网页端登录状态。
  - 浏览器优先级固定为：agent 内置浏览器（Codex in-app Browser） → Codex/Cursor 插件集合系统浏览器（用户的 Edge/Chrome 等系统浏览器） → standalone Playwright 或持久化 Playwright profile。
  - 首选 agent 内置浏览器时，直接使用 Browser 插件的 `iab` 标签页打开/复用携程和同程；如果用户已在内置浏览器登录，优先在同一 profile 中完成搜索和采集。
  - 只有当内置浏览器不可用、未登录且用户需要系统浏览器登录态、被登录/验证码阻断，或用户明确要求复用系统浏览器时，才切到 Codex/Cursor 插件集合系统浏览器；使用插件通道认领或新开用户当前 Edge/Chrome tab，并用 `tab.playwright` 的 locator/DOM 路径操作页面。
  - Playwright 只作为第三优先级 fallback：用于前两种浏览器表面不可用，或需要独立持久化 profile 做复核时。
  - 网页入口优先使用平台真实首页：携程 `https://www.ctrip.com/`，同程 `https://www.ly.com/hotel`；可使用页面生成的结果 URL 复现同一可见用户路径。
  - 使用 Playwright 时，遵循当前会话可用的 Browser/Playwright skill 或插件文档中的认证、locator、等待和多标签页建议。
  - 如果用户明确要求全局 Edge/Chrome profile，且该 profile 已被现有浏览器实例占用，先尝试通过插件集合系统浏览器打开新 tab；不能复用时，再启用一个新的可持久化 profile，不要强行抢占或关闭用户现有浏览器。
  - 如果携程/同程网页因未登录只显示“登录看低价”、跳登录页或无法采集价格，在当前优先级选中的浏览器里打开可见登录页让用户登录；登录后必须复用同一浏览器/profile 继续搜索，不要换浏览器。
  - 用户未登录或暂不登录时，网页端可用于验证日期、房型、评分、设施和可订状态，但价格字段要标记“需登录复核”。
- 本地命令：`node`、`npx`、`curl`、`jq`。

环境修复规则：

- 7 个主来源中，因 key、配置、CLI 缺失、MCP 未暴露、浏览器不可用、未登录等环境问题导致不可用的来源超过 2 个时，必须优先修复环境问题；除非用户明确说“不管/降级继续/跳过这些来源”，不要直接给最终推荐。
- 可自动修复且低风险的项目（例如安装 FlyAI CLI、用 HTTP fallback 验证 MCP、重试网页端、打开网页登录页）应先尝试修复，再报告结果。
- 网页端偶发加载失败可重试一次；重试仍失败时记录失败，不编造结果。
- 若来源服务可调用但返回 0 条候选，记录为空结果并调整查询锚点重试；空结果不计入环境失败数。

## Source Registry

主来源：

- 高德地图 MCP：定位 POI、查酒店/民宿分布、判断住宿锚点、测距和交通可达性。
- FlyAI / 飞猪：通过 `flyai-cli` MCP 或 `flyai` CLI 做结构化酒店搜索，采集价格、评分、图片、地址、预订链接。
- AIGoHotel MCP：若当前会话暴露可调用工具，则采集酒店专源结果；否则优先尝试 Streamable HTTP fallback 或 `scripts/aigohotel-mcp.mjs`，仍失败才报告不可调用。
- 同程程心：官方同程自然语言酒店查询。
- 携程问道：官方携程自然语言酒店查询。
- 同程网页端：按“agent 内置浏览器 → Codex/Cursor 插件集合系统浏览器 → Playwright”优先级模拟真实用户操作，验证同程价格和房源。
- 携程网页端：按“agent 内置浏览器 → Codex/Cursor 插件集合系统浏览器 → Playwright”优先级模拟真实用户操作，验证携程价格和房源。

已排除来源：

- Jinko MCP：v1 不纳入。
- `travel-cn`：v1 不纳入。
- 去哪儿网页端：已验证国内酒店入口和城市页会回主站，移动端猜测入口同样回主站；境外酒店页可打开但搜索跳登录页，不能稳定采集国内酒店列表，v1 不纳入。

## Workflow

1. 读取用户输入，补齐查询参数。
2. 执行 Preflight，输出来源健康检查；遇到配置问题时先请用户处理或确认降级。
3. 用高德定位目的地或 POI，得到坐标、行政区、周边酒店/民宿分布。
4. 选出 1-3 个住宿锚点，例如景区入口、游客中心、换乘中心、停车场、餐饮聚集点、车站或主路节点。
5. 按同一日期、人数、房间数、偏好并行采集各来源。若当前环境和用户授权允许子 agent，按“来源级子 agent 调度”拆分；若不能派发子 agent，则至少使用并行工具调用，并按高德、FlyAI、AIGoHotel、程心、问道、同程网页、携程网页顺序串行补齐缺口。
6. 归一化字段、去重、合并同酒店跨平台结果。某酒店被多个独立来源命中时，在“来源覆盖/受欢迎度/可信度”上加分，并优先做跨平台比价；不要只比较单来源里最便宜的酒店。合并时保留平台链接，优先输出携程/同程详情页。
7. 用高德测距和出行方式修正位置分；检查价格异常和房源异常。
8. 输出报告，不隐藏来源失败和价格冲突。

详细调用步骤见 `references/source-playbooks.md`。归一化、去重和评分规则见 `references/normalization-and-scoring.md`。

## 来源级子 agent 调度

当用户明确要求使用子 agent、并行 agent 或当前运行环境允许来源级委派时，优先使用子 agent 加快采集。目标是“一组互不依赖的来源一个子 agent”，主 agent 负责 Preflight、任务拆分、最终归一化、去重、比价和推荐。

推荐分工：

- `地图/测距 agent`：高德 POI 定位、住宿锚点、候选酒店坐标和自驾/步行/公交距离。
- `FlyAI/AIGoHotel agent`：FlyAI/飞猪与 AIGoHotel 搜索、环境 fallback、结构化价格和房源字段。
- `程心/问道 agent`：同程程心和携程问道自然语言结果，保留原始预订链接和价格文本。
- `网页端 agent`：同程网页端和携程网页端真实浏览器路径。由于会共享 Edge/profile/tab，默认只开 1 个浏览器 agent 串行跑两个网页来源；不要让多个子 agent 同时操作同一个可见浏览器、关闭共享 tab 或抢占登录流程。
- `复核/比价 agent`（可选）：当主 agent 已选出 3-8 个候选酒店后，按酒店名跨来源复核同店价格、房型、取消政策、是否售罄和位置异常。

子 agent 上限与排队：

- 如果可用子 agent 数量小于推荐分工数量，按优先级排队：`地图/测距` → `网页端` → `FlyAI/AIGoHotel` → `程心/问道` → `复核/比价`。
- 已运行的子 agent 不要重复做同一来源；排队任务等前一个 agent 完成或释放名额后再派发。
- 浏览器来源即使排队也不要并发抢同一个 Edge profile；网页端 agent 完成并释放/保留 handoff tab 后，再允许后续网页复核。
- 子 agent 只采集和总结事实，不写 skill 源码、不改共享文件、不做最终推荐。最终推荐必须由主 agent 合并所有来源后给出。
- 如果子 agent 不可用，报告中注明“未使用子 agent，改用并行工具/串行补齐”，但仍按 7 个主来源尽量完成采集。

## 出行方式权重

默认权重：价格 30%、位置 25%、评分 18%、评价/销量 12%、设施/房源可信度 15%。用户明确说“预算有限/性价比/别太贵/价格敏感”时，价格可提高到 35%，从评分和设施中各挪一部分；用户明确说“舒适/高端/亲子设施优先”时，价格可降到 25%。

价格不要先框定一个默认价位。除非用户给出预算上限，先采集本次日期、人数、区域和房型可比的有效价格，按从低到高排序，计算价格分布和 P40 舒适线。低于或等于 P40 的候选应获得价格加分；高于 P40 的候选开始按分位段减分，除非位置、评分、房型或稀缺性足以解释溢价。异常低价仍要复核房型、日期、取消政策和位置，不要把可疑最低价直接当最佳。

按出行方式修正：

- 自驾/有车：提高停车、道路可达、距主路/停车场权重；适度降低离景区入口绝对距离的惩罚。
- 公共交通：提高游客中心、公交/接驳站、步行距离权重；明显惩罚远离交通节点的酒店。
- 打车/包车：保留位置权重，但弱化公共交通节点；强调道路可达、上下车便利、距目标区域的通行距离。
- 普通自由行未指定：使用默认平衡权重，并在报告中说明“按普通自由行默认”。

## 报告结构

按下面结构输出：

```markdown
# [目的地] 酒店推荐

## 来源健康检查
| 来源 | 状态 | 说明 |

## 位置判断
- 目标点：
- 推荐住宿锚点：
- 出行方式影响：

## 价格分布
- 有效价格样本：
- 低价段 / P40 舒适线 / 中位价 / 高价段：
- 本次价格取向：

## 各来源结果
| 来源 | 命中数 | 价格区间 | 字段完整度 | 备注 |

## 合并比价表
| 酒店 | 详情/预订链接 | 推荐区域/距离 | 平台价格 | 来源覆盖 | 评分/评价量 | 设施/房源 | 异常标记 |

## 推荐结论
1. [酒店名](携程或同程详情页)：[推荐理由、适合人群、预订注意]

## 谨慎项
- [酒店名或来源]：[原因]
```

价格评分以本次可比候选的分布为准：P40 是默认舒适线；P40 以下加分，P40-P60 小幅减分，P60-P75 明显减分，P75 以上仅在位置/设施/评分有强理由时推荐。若价格低于同组中位价 35% 或高于同组中位价 50%，额外标记为价格异常，并复核房型、是否无窗/不可取消/非同日期/偏远/房源不足。

## Browser Rule

网页端只做真实用户路径模拟：打开平台酒店页，输入目的地，选择可见联想项，设置日期和人数，点击搜索，采集页面可见酒店名、价格、评分、距离、标签、房源状态和链接。浏览器优先级固定为：agent 内置浏览器（Codex in-app Browser） → Codex/Cursor 插件集合系统浏览器（用户 Edge/Chrome 等系统浏览器） → Playwright MCP/standalone Playwright。可使用页面生成的可见 `hotellist` / detail URL 复现同一用户路径，但不要默认逆向接口、抓隐藏 API 或伪造网页结果。

如果真实用户路径需要登录才能继续或才能看到价格，在当前优先级选中的浏览器里打开可见登录页让用户登录，并说明“登录后我会复用同一 profile 继续查”。登录完成后继续使用同一浏览器/profile 采集；如果用户不登录，继续采集非价格字段并在报告中明确“价格需登录复核”。
