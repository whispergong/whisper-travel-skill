# 完整旅行规划

用这个 skill 做旅行规划时，目标不是写泛泛攻略，而是交付一份能执行的路线：为什么这样走、每天怎么走、路上看什么、住哪里、大概要花多少钱、哪些信息需要复核。

v1 强能力边界：中国境内路线。中国境内用小红书、中文网络攻略、高德 MCP、酒店聚合和费用估算闭环。海外目的地也可以规划，但要用当地语言加英文搜索，并说明高德路线验证不可作为海外主依据。

## 输入要求

先提取：

- 出发地：城市、机场、车站或详细地址。
- 目的地或方向：终点、区域、景区、环线主题；如果只说“云南/川西”等大区域且无法判断核心目的地，追问。
- 出行方式：自驾、公共交通、火车/飞机接驳、包车、骑行等。缺失时追问。
- 可选：出行日期、可用天数、人数、预算、偏好、住宿要求、同行人、是否往返、是否避开高速/夜路/高反/拥堵。

如果没有指定天数，不要先追问天数；先根据攻略常见线路、总里程、交通方式和每日强度推断推荐天数，并说明依据。如果出发地、目的地/方向或出行方式缺失，先简短追问。

## Preflight

开始前做来源健康检查，只输出状态和原因，不回显完整 key 或 cookie：

- 小红书：`xiaohongshu-skills` 是第三方依赖，只调用不修改。先定位已安装项目（优先使用 `XIAOHONGSHU_SKILLS_HOME`，其次当前会话技能列表里的 `xiaohongshu-skills` 路径，再检查 `%USERPROFILE%\.codex\skills\xiaohongshu-skills`、`%USERPROFILE%\.agents\skills\xiaohongshu-skills`、`$HOME/.codex/skills/xiaohongshu-skills`、`$HOME/.agents/skills/xiaohongshu-skills`；仍找不到时请用户提供路径）。检查该项目的 `SKILL.md`、`scripts/cli.py`、`.venv`/`uv`/`python` 和 XHS Bridge。小红书必须走浏览器自动化路径：`scripts/cli.py` 只是通过 XHS Bridge 控制真实 Edge/Chrome 页面，不是绕开浏览器的接口。先进入小红书项目目录再运行命令：Windows 优先 `.\.venv\Scripts\python.exe scripts\cli.py check-login`，类 Unix 优先 `./.venv/bin/python scripts/cli.py check-login`；没有虚拟环境时用 `uv run python scripts/cli.py check-login`，最后才用系统 `python`/`python3 scripts/cli.py check-login`。不要在当前旅行 skill 仓库误跑相对 `scripts/cli.py`。识别浏览器时以 `ws://localhost:9333` 的 XHS Bridge 连接状态为准：若扩展已连接，直接使用当前连接的 Edge/Chrome；若未连接，优先复用用户已安装 XHS Bridge 的浏览器，能判断 Edge/Chrome 时用已安装且已登录的那个，不能判断时请用户打开/启用对应浏览器扩展。未登录、扫码验证或风控提示时，让用户在同一浏览器里处理，再继续。
- 图片读取：确认当前 agent 可以下载图片到本地并用视觉能力读取；若当前环境无法直接读图，使用 `codex-vision-delegation` 或让用户提供可读图片路径。不要忽略小红书图片。
- 高德 MCP/API：确认 `maps_geo`、`maps_direction_driving`、`maps_distance`、`maps_around_search` 等可用；若用户提供、运行时环境存在高德开放平台 key，或 Codex/Cursor MCP 配置里已有 `AMAP_MAPS_API_KEY`，优先把它作为高德 WebService / 静态地图 / JS API 的运行时 key 使用。不要把 key 写进仓库、文档正文、截图、日志摘要或最终回复。
- 酒店：确认 `hotel-search-cn` 存在；多日住宿会由它完成来源健康检查和比价。
- 辅助旅行来源：FlyAI、同程程心、携程问道、普通网页搜索是否可用。
- 腾讯文档：确认当前会话可用的 `tencent-docs` skill 或 `TENCENT_DOCS_SKILL_HOME`、`mcporter`、`tencent-docs` 工具列表、`create_smartcanvas_by_mdx` 和 `upload_image` 可用；若 token 失效或需要授权，先引导用户完成腾讯文档授权，再创建智能文档。
- 腾讯文档额度：优先使用 `tencent-docs` skill/MCP，但把读取、图片上传和正文写入都当作有成本操作管理。先用 1 次 `get_content` 或分页 `smartcanvas.read` 定位当前结构，在本地生成完整 MDX 补丁并自检，再用少量 `smartcanvas.edit` 批量插入/替换；避免边想边多次小改。MCP 因额度、权限或接口异常不可用时，再降级到浏览器编辑。`smartcanvas.read` 分页结果可能在每页开头重复输出页面 frontmatter，验证残留时要先按页剥离这段元信息，避免误判并无意义地反复 UPDATE。
- 通用浏览器优先级：除小红书外，任何需要打开网页、登录页、地图页或最终文档做样式复核的动作，都按 `agent in-app Browser -> 浏览器插件控制 Chrome/Edge 或当前系统浏览器 -> Playwright/持久化 Playwright profile -> Computer Use 兜底`。Codex/agent 自带浏览器若出现临时连接、空 tab、导航或截图小问题，要先按对应浏览器 skill 做轻量恢复尝试：列出/复用当前 tab、新建 tab、reload、先导航到空白页再打开目标、等待/重连、再次截图或导航；至少做数次低成本尝试并记录失败原因，不要一次失败就跳到后级。只有前两者不可用、被拦截或不适合当前任务时，才使用 Playwright；Computer Use 仅用于前述浏览器表面都不可用或必须操作系统可见 UI 时。小红书仍以 XHS Bridge 已连接的真实 Edge/Chrome 为准。
- 本地命令：`python3`、`node`、`curl`、`jq`、`rg`。

如果小红书、图片读取或高德这三个核心能力有两个以上不可用，先修复或请用户处理；除非用户明确降级继续，不要直接输出最终行程。

## Workflow

1. 明确出发地、目的地/方向、交通方式和关键约束。
2. 执行 Preflight，必要时先处理登录、bridge、MCP 或 key 问题。
3. 并行搜集攻略：小红书为高权重来源，但小红书浏览器详情采集要限速且单 agent 串行；携程/同程网页端跟团游、私家团、包车游、当地参团产品也是高权重路线证据，权重与小红书同级；普通网页和当地语言搜索可并行补充；读取小红书图片和重要网页图片。
4. 从文本和图片中抽取候选路线、沿途点、避坑、季节路况、费用、住宿区域和时间建议。
5. 用高德 MCP/API 验证起终点、途经点、每日驾驶/步行/公交可达性、景区周边 POI、住宿锚点和绕路成本。高德相关需求优先走 API：自驾日先用 WebService 路径规划（优先 `show_fields=polyline`、`waypoints`、距离/时长/道路步骤）生成可复核路线数据，再用高德静态地图 `paths`/`markers`/`labels` 或 JS API `AMap.Driving` 渲染实际道路路线图。静态地图里不要只留下 `A/1/Z` 这类无语义标记；`markers.label` 只放数字、大写字母或单个中文字，完整地名/地名简称用 `labels`（不超过 15 字）或文档图例补足；如果静态地图返回图未显示 `labels`，用本地图片后处理按已验证坐标叠加真实地名后再上传。生成路线图时必须按完整 polyline 和所有标注点计算视野范围，保留充足外边距，避免起终点、途经点或地名贴近图片边缘；读图确认路线完整、地名可读、没有顶部白框/异常空白或关键线路裁切后再上传。只有 API 不可用或 key/权限受限时，才按通用浏览器优先级降级：agent in-app Browser -> 浏览器插件控制 Chrome/Edge 或当前系统浏览器 -> Playwright 截图 -> Computer Use 兜底截图。
6. 生成 1-3 条大体路线，按风景密度、沿途有趣程度、驾驶强度、住宿可行性、预算和用户偏好评分，选主方案并保留备选。往返路线尽量避免高度重复；如果重复不可避免，要说明原因并给出可选绕行/拆分方案。用户提出新增景点、合并/拆分天数、换交通方式或调整住宿点时，也必须先用小红书/携程/同程线路、官方信息、普通网页和高德 MCP/API 独立复核可行性、安全性、常规玩法和替代方案；不要盲从用户建议，只有调研和路线核验支持后才写入最终文档。携程/同程产品要记录日程模式、销量/出游人数、评分、评价数、供应商和限制条款；多个高销量高评分产品重复出现的线路模式，可作为强证据。
7. 将主路线拆成每日计划；完整旅行攻略的每日具体线路必须按二级或三级标题逐日展开，不要用一个大表格承载所有每日细节。每一天包含“今日动线/路线流”、推荐途经点、景点/官方介绍文字超链接、高德 MCP 核验的坐标/驾车距离/时长依据、路线图或路线示意、驾驶/停车/天气/补给提醒、重点预约/路况复核和当晚住宿策略。新增、删除、合并或拆分天数时，先维护一份本地“天数账本”（Day、标题、起终点、住宿、路线图、后文引用），再一次性更新总览、每日标题、图片 alt/说明、住宿/预约/费用公式和后文 DXX 引用；可以在本地草稿用有序列表、映射表或脚本辅助自动编号，但最终腾讯文档仍要展示稳定清晰的 `D1/D2/...` 标题，不使用读者可见的公式或有序列表编号来替代 Day 标题。景点预约、官方介绍、小红书攻略和酒店链接都应使用说明文字作为超链接，不要在最终文档裸露长 URL；写入腾讯智能文档时优先使用标准 Markdown 链接 `[说明文字](URL)` 并读回验证 href，避免使用会丢 URL 的未验证 `<Link>` 组件。
8. 路线稳定后，为每天住宿点调用 `hotel-search-cn`，传入日期、人数、交通方式、住宿锚点和偏好；要求住宿子流程保留酒店详情页/预订页链接，优先携程和同程，其次飞猪或来源搜索页。
9. 汇总费用表：酒店、油费、过路费、门票、停车/接驳、餐饮、备用费用；区分实价、攻略价和估算。用户未指定餐饮标准时，默认按 `人数 × 天数 × 75 元/人/天` 估算，并把人数和天数显式写进公式。
10. 输出报告，不隐藏来源失败、路线不确定性、季节/天气/路况风险和价格需复核项。
11. 报告稳定后，使用 `tencent-docs` 创建一篇腾讯智能文档作为最终交付；文档要美观、可分享，面向最终用户阅读。最终文档包含标题、高德核验路线总览图、尽量覆盖每天的路线图或路线流程图、每日独立章节、带超链接的具体酒店候选、景点介绍链接、大景点的小红书游览攻略链接、住宿和费用表；路线图优先使用高德 WebService polyline + 静态地图或高德 JS API 截图形成的实际道路路线图，并突出起点、推荐途经点/补给点和终点，图上应尽量显示真实地名或地名简称。最终文档中的图片 alt、图片下方说明和段落标题要面向旅行者自然描述，例如“D7 大柴旦到茫崖自驾路线图”，不要暴露 `WebService`、`静态地图`、`含地名标签`、调试方式或工具流水。若每日图不是高德真实页面、静态地图或真实地图截图，必须基于高德驾车规划步骤/道路/分段数据生成导航式示意，并用自然文案说明它是路线示意，不要冒充实时导航截图；不要把来源健康检查、工具失败日志、调试过程写进文档正文。若生成封面、氛围图或非地图图片，明确优先使用 `GPT-IMAGE2`。
    - 更新已有长文档时，采用“先本地合并、后批量写入”的省额度策略：一次性定位锚点和待删除范围；图片先批量上传并复用 `image_id`；正文按稳定章节块写入，避免逐句编辑；写入后只做一次读回验证，再针对失败项做最小补丁。保持既有文档大纲稳定，不为“调整说明”“版本记录”“某段重排”临时新增大纲级章节；把新增景点、拆分理由、路线风险和来源依据写入对应 Day 的每日段落、预约/复核或驾驶提醒中。
    - 链接优先用标准 Markdown `[说明文字](URL)`；如果腾讯 SmartCanvas 读回出现 `[]()` 空 href 或长 URL 裸露，改用单行 MDX `<Link href="URL">说明文字</Link>` 组件，并再次读回确认。若 Markdown 和 `<Link>` 两种写法读回/浏览器视图仍无法保留外链，但最终页面没有裸露长 URL，则停止重复 MCP 试写，保留说明文字、关键词和复核入口，并在交付说明中标注“链接 href 需在腾讯文档端二次确认或手动补齐”。
    - MCP 读回通过后，仍必须按 `references/tencent-smart-doc.md` 的 Browser Visual Verification 打开最终文档做一次真实页面样式检查，确认首屏、路线图、每日图片、图注、链接和表格宽度在腾讯文档里实际可读。

详细来源调用见 `references/source-playbooks.md`。路线和费用规则见 `references/route-and-costing.md`。并行/子 agent 调度见 `references/agent-orchestration.md`。腾讯智能文档交付见 `references/tencent-smart-doc.md`。

## Guardrails

- 不要只凭单篇攻略定路线；至少尝试小红书 + 一个补充来源 + 高德验证。
- 不要把图片当装饰；图片里的路线截图、景区地图、菜单价目表、停车信息和路况提示都要读取并转成事实。
- 不要把小红书当普通 HTTP/API 抓取源；它是浏览器自动化来源。不要让多个 agent 同时操作小红书详情页或同一个可见浏览器。
- 不要用直线距离替代自驾/公共交通时间；能用路线规划时必须用路线规划。
- 不要让酒店子 agent 决定最终总路线；酒店结果用于确认住宿可行性和预算。
- 不要只把攻略留在聊天里；除非腾讯文档不可用或用户明确不要，最终必须创建腾讯智能文档并返回 URL。
- 不要把聊天里的来源健康检查复制进腾讯智能文档；它是执行诊断，不是给最终用户看的攻略内容。
- 不要用泛泛的装饰图替代路线图；总览图和每日路线图必须优先基于高德 API/MCP 核验的路线规划步骤、道路、polyline、路段或真实地图截图。高德 WebService/静态地图/JS API 可用时，优先 API 实现；浏览器截图只是 API/key/权限不可用后的兜底，且必须按通用浏览器优先级执行；不要用简单坐标直线图替代每日自驾路线图，也不要只交付 `A/1/Z` 这类无法直接读懂的点位标记，至少补充真实地名标签、地名简称或清晰图例。更新既有文档的路线图时，先观察已完成日程（例如 D1/D2）的路线图样式和真实地图来源；新增或重排的日程图必须维持同等真实地图/高德路线图级别，不得把后续新图降级成纯示意图或坐标折线图。
